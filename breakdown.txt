CHARTLY — WHAT WE'VE BUILT SO FAR
==================================

This file is a plain-English walkthrough of the project. Read it top
to bottom; each section builds on the last.

------------------------------------------------------------
0. WHAT IS CHARTLY?
------------------------------------------------------------

Chartly is a REACT LIBRARY — code that OTHER developers install into
THEIR React apps. It's not a website. It's not an app you run. When
finished, someone will do:

    npm install chartly

    import { LineChart, Line } from "chartly";

...in their own project, and use the chart components we've written.

That distinction — "library, not app" — drives most of the weird
decisions in this project. A library has to:

  - Publish to npm as a package
  - Ship in TWO formats (ESM and CJS — explained below) so both
    modern bundlers and older Node.js can use it
  - Ship .d.ts TypeScript type files so consumers get autocomplete
  - Externalize React (not bundle it) — the consumer already has React
  - Be tiny — nobody wants a charting lib that adds 500KB

A React APP does none of that. It bundles everything, ships to a
website, and never gets imported.

------------------------------------------------------------
1. WHY IS `node_modules/chartly` A SYMLINK?
------------------------------------------------------------

You noticed `node_modules/chartly` exists. That's not a mistake.

We're using an npm feature called WORKSPACES. It lets one repository
hold multiple packages. Look at the folder layout:

    chartly/
    ├── package.json           <-- the "root" — declares workspaces
    ├── .gitignore             <-- files git should ignore (see below)
    ├── breakdown.txt          <-- this file
    ├── .vscode/
    │   └── settings.json      <-- workspace TypeScript version pin
    ├── node_modules/
    │   └── chartly -> ../packages/chartly    <-- SYMLINK (shortcut)
    └── packages/
        ├── chartly/           <-- the actual library
        └── docs/              <-- (empty for now — Phase 9)

  What .gitignore does: git normally tracks every file in the repo,
  which is bad for auto-generated stuff like `node_modules/` (millions
  of files, hundreds of MB, reproducible from package.json) and `dist/`
  (the built output, rebuildable from src). .gitignore is a plain-text
  list of paths git will pretend don't exist. Ours excludes:

    - node_modules/            (deps — reinstall with npm install)
    - dist/, build/, out/      (auto-generated builds)
    - coverage/                (test coverage reports)
    - *.log                    (npm/yarn debug logs)
    - .env, .env.*             (secrets — NEVER commit)
    - .DS_Store, Thumbs.db     (OS-generated junk)
    - .idea/, .vscode/*        (editor prefs — with an exception:
                                 we DO commit .vscode/settings.json
                                 because it pins the workspace TS
                                 version for everyone; personal
                                 launch configs etc. stay local)
    - .cache/, .vite/          (dev-tool caches)
    - image.png, image-*.png   (reference screenshots you drop in
                                 for design chats — useful locally,
                                 don't ship to git)

  The `!` prefix in a gitignore is a re-include: `!.env.example` means
  "even though .env* is ignored, DO track .env.example". Useful for
  template files that document what env vars a real .env needs.

When you run `npm install` in the root, npm looks at the root's
`workspaces` field:

    "workspaces": ["packages/*"]

...and creates a symbolic link inside `node_modules/` for each
package in `packages/`. So `node_modules/chartly` is literally a
shortcut that points back at `packages/chartly`.

WHY WE WANT THIS: eventually `packages/docs/` will be a demo site
that IMPORTS chartly (like a real user would). Because of the
symlink, when docs does `import { LineChart } from "chartly"`, it
finds our in-development version — no publishing needed.

You can safely ignore `node_modules/chartly`. It's just plumbing.

3. PHASE 1 — WHAT WE ACTUALLY DID (SCAFFOLDING)
------------------------------------------------------------

Phase 1 built the "workshop" — no chart code yet, just the pipeline
that lets us write TypeScript and produce a publishable npm package.

Six files were created/rewritten. Here's what each one does and WHY
it exists:

--- `tsconfig.json` ------------------------------------------
TypeScript's config file. Tells the compiler:
  - "target": "ES2020"     — output modern JS (last-5-years browsers)
  - "jsx": "react-jsx"     — use the modern JSX transform (no need
                             for `import React from "react"` in every file)
  - "strict": true         — turn on all safety checks
  - "noEmit": true         — DO NOT actually output files. tsup will.
                             tsc is just here to CHECK the types.
  - "moduleResolution": "Bundler"  — resolve imports the way modern
                             bundlers (tsup, vite) do
  - "ignoreDeprecations": "6.0"  — silence a warning about `baseUrl`
                             that tsup's DTS builder triggers

--- `tsup.config.ts` ----------------------------------------
tsup is a bundler — it takes our TypeScript source and produces
distributable JavaScript. Config says:
  - entry: "src/index.ts"   — start from our public entry
  - format: ["cjs", "esm"]  — output BOTH formats (see next section)
  - dts: true               — also generate .d.ts type files
  - external: ["react", "react-dom"]  — DO NOT bundle React

  Why external React? If we bundled it, every consumer's app would
  have TWO copies of React — ours and theirs. React breaks if there
  are two copies (hooks fail, hydration errors). So we require the
  consumer to provide React, and just import from it.

--- `package.json` -------------------------------------------
This is the file npm reads when someone installs "chartly". Key
fields we set:

  "type": "module"    — this package is ESM by default
  "main":    "./dist/index.cjs"  — if a CJS user requires "chartly",
                                    give them this
  "module":  "./dist/index.js"   — if an ESM bundler imports it, give
                                    them this
  "types":   "./dist/index.d.ts" — TypeScript, use these type defs
  "exports": { ... }             — modern per-condition routing (the
                                    "exports" field is more precise
                                    than main/module — it lets us
                                    map different files per situation:
                                    import vs require, types vs code)
  "files": ["dist"]  — only publish the `dist/` folder to npm.
                       Nobody wants our source, only the built output.
  "sideEffects": false  — tells bundlers "our code has no top-level
                          side effects, feel free to drop unused
                          exports" (enables tree-shaking)
  "peerDependencies": { react }  — see React-external explanation above

  Scripts:
    "build"     : "tsup"            — produces dist/
    "dev"       : "tsup --watch"    — rebuilds on file change
    "typecheck" : "tsc --noEmit"    — check types without emitting
    "test"      : "vitest run"      — run tests once

--- `src/index.ts` -------------------------------------------
The PUBLIC ENTRY POINT. Only things exported from here are visible
to library consumers. It's the "table of contents" for what users
can import.

--- `vitest.config.ts` --------------------------------------
Test runner setup. Uses `jsdom` (a fake browser DOM that runs in
Node) so React component tests can pretend they're in a browser.


------------------------------------------------------------
3a. QUICK DIGRESSION: ESM vs CJS (WHY WE SHIP BOTH)
------------------------------------------------------------

There are two ways to write module code in JavaScript, and both are
in the wild:

  ESM (modern, since ~2020):
      import { foo } from "bar";
      export const baz = ...;

  CJS (older, since Node ~2010):
      const { foo } = require("bar");
      module.exports.baz = ...;

Modern apps (Next.js, Vite, ESM-native Node) use ESM.
Older Node.js scripts and some tooling still use CJS.

If we only shipped ESM, CJS users would get errors. If we only
shipped CJS, we'd miss out on tree-shaking optimizations for modern
bundlers. So we ship BOTH — that's what `format: ["cjs", "esm"]`
in the tsup config does. The `exports` field in package.json
routes each consumer to the right file.

That's what "dual-format package" means. It's the standard for
modern npm libraries.

4. PHASE 2 — WHAT WE ACTUALLY DID (SCALE FOUNDATION)
------------------------------------------------------------

Phase 1 built the workshop; Phase 2 started building the actual
library. But we deliberately built NO VISUALS yet — just the
math and structure. Here's the reasoning: every chart type
we'll build later (Line, Bar, Area, etc.) needs the same three
things:

  1. A way to know how big it is (responsive sizing)
  2. A way to convert data values into pixel positions (scales)
  3. A way to share those with child components (context)

If we build those three things ONCE, correctly, every chart in
Phases 3–5 just plugs into them. If we build them wrong, we redo
9 chart types. So Phase 2 is boring but load-bearing.

--- What "scale" means ---------------------------------------
A "scale" is the D3 term for a function that maps DATA to PIXELS.
Example: your data has values from 0 to 100, but your chart is
400 pixels wide. A linear scale takes value 0 -> pixel 0, value
100 -> pixel 400, value 50 -> pixel 200. That's it.

Three scale types matter for Cartesian charts:
  - linear:  numbers   (revenue, temperature, counts)
  - time:    dates     (timestamps for time series)
  - band:    categories (product names, days of week)

D3 provides these; we just wrap them nicely.

--- Files we created in Phase 2 ------------------------------

`src/types.ts`
  Public TypeScript types shared everywhere:
    - Margin           — { top, right, bottom, left } object
    - DEFAULT_MARGIN   — sensible defaults
    - ScaleType        — "linear" | "time" | "band"
    - ScaleInput<S>    — magic type: given a scale type, tell me
                          what data values it accepts. E.g. if S is
                          "time", ScaleInput<S> is Date.
    - KeysOfType<T, V> — magic type: given an object type T and a
                          value type V, list the keys of T that have
                          values of type V. Powers the `xKey` prop.

`src/scales/types.ts`
  Internal aliases: XScale/YScale = union of d3-scale's return types.

`src/hooks/useResize.ts`
  A custom React hook that uses ResizeObserver (browser API) to
  measure the current size of a DOM element. Returns { ref, width,
  height }. The consumer attaches `ref` to a div; the hook watches
  that div and updates width/height whenever it resizes.

  Why: we decided NO fixed width/height props. Instead the chart
  fills its parent. This hook is HOW we do that.

`src/scales/useChartScales.ts`
  The heart of the phase. Takes:
    - data
    - accessor functions (how to extract x/y values from each row)
    - scale types
    - margin
    - current width/height

  Returns:
    - xScale and yScale (D3 scale functions ready to call)
    - innerWidth/innerHeight (the drawable area, minus margins)

  Critical detail: the pixel RANGE bakes in margin.
    x range: [margin.left, width - margin.right]
    y range: [height - margin.bottom, margin.top]   <-- INVERTED

  Why inverted for y? Because in SVG, y=0 is at the TOP of the
  screen and increases downward. But charts show high values at the
  top. So we flip the range to compensate. Every chart gets this
  for free from the scale — no per-chart math.

`src/context.ts`
  React Context is a way to share values with all descendants
  without passing props at every level. We create a `ChartContext`
  holding scales/margins/dimensions, and a `useChartContext()` hook
  for children to read it.

  Why: our architecture says "context + hooks for prop sharing".
  When someone writes <ChartContainer><Line /></ChartContainer>,
  the Line component doesn't receive scales as a prop — it grabs
  them out of context via useChartContext(). Cleaner API.

`src/ChartContainer.tsx`
  The React component consumers will use. Roughly:

    1. Get the container's size via useResize()
    2. Fill in default margin
    3. Turn xKey/x and yKey/y props into accessor functions
    4. Call useChartScales() to build scales
    5. Render <div><svg>{children}</svg></div>
    6. Provide the scales via ChartContext to whatever children
       the user passed in

  The tricky bit is the prop types. We allow two ways to say
  "which field is x?":

    <ChartContainer xKey="date" ... />         // by field name
    <ChartContainer x={(d) => d.date} ... />   // by function

  These are combined into a discriminated union in TypeScript so
  you can use EITHER — but not both — and the type system checks
  that xKey (if used) actually points at a value type matching the
  chosen scale (e.g. "time" scale requires a Date value).

`src/__tests__/useChartScales.test.ts`
  Six tests, all passing. The most important is the y-inversion
  test — it locks in the "margin baked into range" decision, so
  a future refactor can't accidentally break it without a test
  failing.

4a. PHASE 3 — WHAT WE ACTUALLY DID (FIRST REAL CHART)
------------------------------------------------------------

Phase 3 is where charts start LOOKING like charts. We built three
components: `<Line>`, `<XAxis>`, `<YAxis>`. Together with the
container from Phase 2, you can now render a real line chart.

Why THIS was the phase that "validates the architecture": three
things had to work end-to-end without duct tape:

  1. A visual component (Line) reads scales from context and
     produces correct SVG geometry.
  2. Axis components independently read the same context and
     produce ticks that line up with the data.
  3. When the container resizes, both Line and axes re-render in
     sync (proves context + ResizeObserver + scales all wire up
     correctly).

If any of those had needed a hack, we'd have redone Phase 2.
Everything worked. So the pattern is now the template for Phases
4–5.

--- The signature customization pattern: RENDER-PROP LABELS ---

Every chart type will eventually let consumers put custom labels
on data points ("$270" above the last dot, for example). There
are many ways to design this API. Options we considered:

  (a) label={{ show: true, format: "%d" }}     -- config object
  (b) labelKey="revenue"                       -- a data field name
  (c) label={(props) => <text>...</text>}      -- a function that
                                                  returns JSX

We picked (c) — the "render prop" pattern. A render prop is just
a function you pass as a prop, and the component calls it and
renders whatever JSX it returns. Why (c) wins:

  - Full flexibility: consumers can style, filter, or hide labels
    per data point by returning null.
  - Fully typed: the function receives {x, y, datum, index} where
    `datum` is your original data row (typed via generics), so
    autocomplete works on your data fields.
  - No custom mini-language: it's just React, so consumers who
    know React already know how to use it.

The Line component takes an optional `label` prop:

    <Line<Row>
      label={({ x, y, datum, index }) =>
        index === lastIndex
          ? <text x={x} y={y - 8}>${datum.revenue}</text>
          : null
      }
    />

That signature (typed { x, y, datum, index } → ReactNode) is now
the standard for every future shape: Bar, Area, Scatter, etc.

--- Files we created in Phase 3 ------------------------------

`src/shapes/types.ts`
  Just two shared types:
    - ShapeLabelRenderProps<T>  — { x, y, datum: T, index }
    - ShapeLabelRender<T>       — (props) => ReactNode
  Every shape's `label` prop uses these.

`src/shapes/Line.tsx`
  Uses `d3-shape`'s `line()` generator to turn data points into
  an SVG `<path>` string. Steps inside the component:
    1. Read data + scales + accessors from context via
       useChartContext().
    2. For each data row, compute a pixel {x, y} by running the
       accessor then the scale (e.g. `xScale(xAccessor(row))`).
    3. Feed those points into `d3.line().x(...).y(...)`, which
       returns a string like "M100,200L120,180L140,190".
    4. Render that as `<path d={...} stroke=... fill="none" />`.
    5. If the `label` prop is set, call it for each point and
       render the returned JSX.

  The `.defined()` filter on the d3 line drops any NaN points
  (safety for undefined data), so we never emit a broken path.

`src/axes/types.ts`
  Shared axis types:
    - TickValue           — number | Date | string (all scale outputs)
    - TickFormatter<V>    — (value, index) => string. Consumer-facing;
                             lets you say "format ticks as $%.2f" etc.
    - AxisProps<V>        — { tickCount?, tickFormatter?, label?,
                             labelOffset?, stroke? }

`src/axes/XAxis.tsx` and `src/axes/YAxis.tsx`
  Both render three things:
    - the axis LINE itself (an <line>)
    - each TICK: a short line + a text label using tickFormatter
    - optionally, a static axis LABEL (the axis TITLE, e.g. "Month")

  How tick values come from: for linear/time scales we call
  `scale.ticks(n)` (D3 returns ~n evenly-spaced sensible values).
  For band scales we use `scale.domain()` directly (every category
  is a tick). YAxis rotates its label -90° so it reads bottom-to-top,
  the standard convention.

`src/__tests__/setup.ts`
  A vitest "setup file" — runs once before all tests. Two jobs:
    - Register @testing-library/jest-dom matchers (adds things like
      `.toBeInTheDocument()` to expect())
    - Stub globalThis.ResizeObserver so ChartContainer doesn't crash
      in jsdom (jsdom doesn't implement ResizeObserver)

`src/__tests__/testUtils.tsx`
  A helper that lets tests skip ChartContainer entirely and render
  Line/XAxis/YAxis inside a hand-crafted ChartContext.Provider.

  Why: ChartContainer needs a real ResizeObserver to know its width.
  In tests we want fixed, predictable dimensions. So we mock the
  context with 400×300 dimensions and known scales, then render the
  component we're testing directly inside it. This is a standard
  React testing pattern for context-consuming components.

`src/__tests__/Line.test.tsx`
  Four tests. The important ones:
    - Line renders exactly one <path> with a valid `d` attribute
    - The label render prop is CALLED once per data point
    - Returning null from label skips that point (proves the
      per-point filter mechanism works)

`src/__tests__/Axes.test.tsx`
  Five tests total across both axes. Covers tick rendering,
  tickFormatter application, and axis label positioning (including
  the YAxis rotate(-90) assertion).

Total test count went from 6 (Phase 2) to 15 (Phase 3).

--- Context extension (a small Phase 2 patch) ----------------

Line needs the resolved x/y accessor functions to compute pixel
positions per datum. Phase 2's context only carried the scales.
So we added two fields to ChartContextValue:

    xAccessor: (d) => number | Date | string
    yAccessor: (d) => number | Date | string

ChartContainer resolves xKey/x and yKey/y into these functions
and puts them in the provider value. Zero public-API change —
context is internal.

--- What "start dev server and see it in a browser" means ----

There's a general Claude Code guideline: for UI changes, start
the dev server and actually look at the feature in a browser
before calling it done. For a library this is awkward — the
library has no "page." The playground solves that: it's a real
page, running the library, that we can look at.

Every phase from 4 onward will use the playground as its "did we
actually build this right?" check.

4b. PHASE 4 — WHAT WE ACTUALLY DID (INTERACTIVITY)
------------------------------------------------------------

Phase 3 charts are pretty but frozen — moving the mouse over them
does nothing. Phase 4 makes them respond. Drop a `<Tooltip>` inside
`<ChartContainer>` and now hovering over the line reveals:

  - A vertical guide line at the nearest x
  - A dot on the nearest data point
  - A card next to the point showing whatever JSX you wrote

Scope was intentionally tight: ONE new component (`<Tooltip>`), ONE
new internal hook (`useHoverState`), no changes to any existing
files. That "purely additive" property is a good smoke test for the
architecture — it means the abstractions from Phases 2–3 are
composable enough to bolt features on without a rewrite.

--- The nearest-point problem: why bisector? ------------------

When the mouse is at pixel x=140, which data point is the "nearest"?
For a chart with 5 points, you could just loop through all of them
and compute distances — that's O(n) per mousemove event, ~60 times
per second. Fine for 5 points. But time-series charts often have
1000+ points, and O(n) at 60Hz starts costing real cycles.

D3 provides a smarter approach in `d3-array`: `bisector`. It's a
binary search on a sorted array (O(log n) instead of O(n)). Here's
what happens per mousemove:

  1. Read mouse pixel x from the event.
  2. `xScale.invert(pixelX)` → the DATA value the mouse hovers.
     (Scales know how to go both directions: data → pixel via the
     scale, pixel → data via .invert().)
  3. `bisector.center(xValues, dataValue)` → the INDEX of the
     nearest data point in the sorted x-values array. Binary
     search under the hood.

Result: even with 100,000 points, nearest-point lookup stays
under a microsecond. This is the specific piece the CLAUDE.md
roadmap called out — because getting it wrong is what turns
"snappy chart" into "laggy chart" on large datasets.

Band scales (categorical x-axis like ["Jan","Feb",...]) can't
use .invert() — there's no meaningful inverse for a category
name from a pixel. So for band scales we fall back to the O(n)
pixel-distance loop. Categories are usually small in count so
this is fine.

--- How the pointer events work ------------------------------

`<Tooltip>` renders an invisible `<rect>` covering the plot area.
That rect has three properties that make it work:

  - `fill="transparent"` — nothing visible on screen, but SVG
    still hit-tests it. Try `fill="none"` and it would let
    pointer events pass through — subtle SVG gotcha we handle
    correctly.
  - `onMouseMove` — fires on every mouse move over the rect.
    Fires the useHoverState hook to update the active point.
  - `onMouseLeave` — fires when the mouse leaves the rect.
    Clears the active state so the tooltip disappears.

Critical detail: the rect is the LAST element in the tooltip's
`<g>`. SVG renders in document order (later elements paint on
top of earlier ones) AND does hit-testing top-down. So by
putting the rect last we ensure:

  - Pointer events land on the rect, not on the guide line/dot
    (which would break onMouseLeave because leaving the dot to
    return to the rect would fire a spurious mouseleave).
  - The `pointerEvents="none"` on the indicator and content
    groups guarantees this even if a consumer's tooltip JSX is
    opaque.

We also require consumers to put `<Tooltip>` LAST inside
`<ChartContainer>` so the tooltip's rect renders on top of the
Line and axes. Documented in the Tooltip type's JSDoc; not
enforced programmatically. This is a small tax for a large
simplicity gain (no z-index-like layering system needed).

4c. PHASE 5a — WHAT WE ACTUALLY DID (CURVES, AREA, GRADIENT)
------------------------------------------------------------

The original Phase 5 in CLAUDE.md was a big lift — 6 chart types
in one shot (Bar, Area, Scatter, Composed, Stacked Bar, Stacked
Area). You showed a reference chart of a curved line with a
green gradient fill, and asked "can my chart achieve this?" — so
we scoped this pass down to the two ingredients that reference
needs: a `curve` prop and an `<Area>` component. Bar/Scatter/
Composed/Stacked become **Phase 5b** later.

--- What a "curve" means in D3 -------------------------------

Given a set of points, there are many ways to draw a line
through them:

  - LINEAR — straight segments between each point (Phase 3
    default; what our Line did before this phase)
  - MONOTONE — a smooth curve that never overshoots. Financial
    charts almost always use this. Your reference used this.
  - STEP — a staircase, e.g. rate stays constant until the next
    reading. Good for step-change data.

D3 provides these as "curve factories" from d3-shape:
`curveLinear`, `curveMonotoneX`, `curveStepAfter`. We wrap them
in a friendly string API:

    <Line curve="monotone" />
    <Area curve="monotone" />

Both components take the same `curve` prop, and both use a shared
resolver in `src/shapes/curves.ts` that maps the string to the
correct d3 factory. Zero surface added on the consumer's side
beyond one prop.

--- The `<Area>` component -----------------------------------

An Area chart is just a filled polygon between the line and a
baseline. D3 provides `d3-shape.area()` — same idea as `line()`,
but takes both a top y (`y1`) and a bottom y (`y0`).

Our Area:
  - Uses `y0 = height - margin.bottom` (bottom of plot area) as
    the baseline. Not the data's y=0, which would be off-screen
    for the reference chart (rates around 0.87). Common charting
    convention.
  - Takes the same fill/stroke/curve/label props as Line.
  - Emits a `<path>` with a closed `d` attribute (starts with M,
    ends with Z). SVG paint fills the enclosed region.

Consumer places `<Area>` BEFORE `<Line>` in JSX so the line paints
on top. Same document-order convention as Tooltip going last.

--- The `<LinearGradient>` helper -----------------------------

SVG gradients are defined via `<defs><linearGradient>...` blocks
and referenced by URL:

    <defs>
      <linearGradient id="fade" x1="0" y1="0" x2="0" y2="1">
        <stop offset="0%" stop-color="#a8ce34" stop-opacity="0.35" />
        <stop offset="100%" stop-color="#a8ce34" stop-opacity="0" />
      </linearGradient>
    </defs>
    <path fill="url(#fade)" ... />

Every phase-5 area chart needs this same 5-line block, so we
wrapped it in a component:

    <LinearGradient
      id="fade"
      from="#a8ce34"
      to="#a8ce34"
      fromOpacity={0.35}
      toOpacity={0}
    />
    <Area fill="url(#fade)" />

That's it. Direction defaults to vertical (top → bottom).
Set `direction="horizontal"` for left → right. Bring-your-own
`<defs>` still works — the helper is opt-in, not a wall.

Why a component (not a hook or a utility function)? SVG gradients
LIVE in the SVG tree — they need to be children of the `<svg>` to
be found by `url(#id)`. A component naturally lives inside JSX and
composes with the rest of the chart. A hook would need to inject
into the DOM tree separately, which is awkward.

4d. PHASE 5b — WHAT WE ACTUALLY DID (BAR CHART)
------------------------------------------------------------

Phase 5b starts delivering the shapes we deferred in 5a. Rather
than doing all 5 remaining shapes at once, we're slicing them
into small phases. First slice: `<Bar>`.

This phase mattered for a specific structural reason: it's the
first shape to use a **band** x-scale. Line/Area have always been
plotted on linear or time scales — continuous x. Bar plots on
categorical x — ("Widgets", "Gadgets", "Gizmos"). Getting Bar
right means the band-scale code path we set up back in Phase 2
finally gets exercised end-to-end: container props →
`useChartScales` → context → shape → tooltip's band-fallback in
`useHoverState`.

If band scales had been broken anywhere in that chain, we'd
have caught it here. They weren't. Good.

--- What a "band scale" is -----------------------------------

Linear/time scales map a CONTINUOUS domain (numbers, dates) to
pixels. A band scale maps a DISCRETE domain (a list of category
strings) to pixels. It divides the pixel range into equal-width
"bands" — one per category — with optional padding between them.

Concretely, given:
  - domain: ["A", "B", "C", "D"]
  - range: [40, 390]  (350 pixels wide)
  - padding: 0.1     (10% gap between bands)

d3's band scale gives you back:
  - scale("A")       → 47 (left edge of A's band)
  - scale("B")       → 132
  - scale.bandwidth() → ~75 (width of every band)

Bar uses `scale(x)` for the bar's left edge and `bandwidth()` for
its width. That's why all bars in a bar chart are the same width
without us having to compute anything ourselves.

--- Bar's geometry -------------------------------------------

For each row of data, Bar computes a `<rect>`:

    x       = xScale(category)              // left edge
    width   = xScale.bandwidth()            // from the band scale
    y       = min(baselinePx, yScale(value))
    height  = |baselinePx - yScale(value)|

The `baseline` is where bars start FROM. The default is `0` — a bar
for value 25 goes from 0 to 25. But if your y-domain doesn't
include 0 (e.g., revenue is 120–270), yScale(0) would produce a
pixel OUTSIDE the plot area. We clamp:

    baselinePx = clamp(yScale(baseline), plotTop, plotBottom)

The clamping keeps bars looking sensible for typical
positive-only datasets: they visually stand on the plot floor.
For datasets that include 0 or negative values, `yScale(0)` sits
naturally inside the plot area and no clamping happens. If you
need to override, pass a `baseline` prop (in data-value terms) —
Bar takes it from there.

--- Fallback for non-band x-scales ---------------------------

If a consumer accidentally uses `<Bar>` inside a
`<ChartContainer xScaleType="linear">`, the x-scale won't have
`.bandwidth()`. Rather than crash, we fall back to a 20 px fixed
bar width. Not idiomatic, but non-fatal — the developer sees
weird-looking bars instead of a white screen and can fix.

--- Tooltip integration — zero code changes ------------------

Phase 4's `useHoverState` already handles band scales: it can't
`invert` on a band scale, so it falls back to a pixel-distance
loop over the data, snapping to the nearest band center. That
means the existing `<Tooltip>` works with Bar with no changes —
drop it in and it snaps to whichever bar you're hovering.

Bar's `label` render-prop passes `x = bandPos + bandWidth/2`
(bar top-center) and `y = top` (top of the bar). Consumers get
consistent geometry across all shapes.

--- Small follow-up: tooltip clipping when data hits the top ---

While playing with the bar chart in the browser, we caught a
visual bug: hovering the TALLEST bar (Gadgets, 65 units) made
the tooltip card overflow the top of the chart card. The top
edge of the "65 units" text was chopped off.

Root cause: our playground tooltip used a fixed offset
`translate(x, y - 46)` — always drawing the card 46 px ABOVE
the active point. For the tallest bar, `y` is very small (bar
top is near the SVG's top edge), so `y - 46` becomes negative
and the browser clips content outside the SVG viewport.

Fix (in playground/App.tsx, both tooltips): FLIP the card
below the point when there isn't enough room above:

    const cardH = 40;
    const gap = 8;
    const above = y > cardH + gap;
    const dy = above ? -cardH - gap : gap;
    return <g transform={`translate(x, y + dy)`}>...</g>;

This isn't a library bug — Chartly gives you the pixel position
of the active point, and it's up to your render function to
decide where the tooltip card sits relative to that. The flip
pattern above is a small recipe every real-world tooltip
eventually adopts. Something to keep in mind when you build
your own tooltip render functions.

4f. PHASE 5d — WHAT WE ACTUALLY DID (COMPOSED CHARTS)
------------------------------------------------------------

A "composed chart" is one that shows more than one series at
once — e.g. bars for monthly revenue with a smooth line overlay
for the running average. All four Cartesian shapes (Line, Area,
Bar, Scatter) technically already worked inside the same
ChartContainer before this phase — but they all read the SAME
`yAccessor` from context, meaning they all plotted the same
column. Useful for redundancy demos, useless for anything real.

Phase 5d fixes that with the smallest possible change: each
shape gains an optional per-shape y accessor. Same data, same
x-scale, same y-scale — but each shape picks which COLUMN of
the data row it draws from.

--- The two flavors of "composed" -----------------------------

There are two ways charting libraries handle multiple series:

  1. SAME DATA, DIFFERENT COLUMNS
     data = [{month, revenue, avg}, ...]
     Bar reads .revenue; Line reads .avg. Shared y-scale.

  2. DIFFERENT DATA ARRAYS PER SHAPE
     bars data = [...]; line data = [...]
     Potentially different x-values, different lengths,
     different y-ranges. Often means a SECOND y-axis on the
     right side of the chart.

Phase 5d ships flavor (1) only. Flavor (2) is a bigger design
question:
  - How does the container know each shape's data?
  - Do we support a secondary y-scale, and if so, how does the
    consumer opt into it?
  - Which series does the tooltip snap to?

Real answers, but deferred until we have a concrete need. In
practice, MOST composed charts fit flavor (1) — the classic
"revenue + running average" or "temperature + rainfall" cases
have all series on the same monthly (or daily) x rows.

--- The `y` and `yKey` per-shape props ------------------------

Every shape now accepts two optional props:

    y?: (d: T) => number    -- accessor function
    yKey?: KeysOfType<T, number>  -- key into T (numeric fields only)

Precedence: if `y` is set, use it. Else if `yKey` is set, use
`d[yKey]`. Else fall back to the container's yAccessor (the
column set via `<ChartContainer yKey="...">`).

Usage in a composed chart:

    <ChartContainer
      data={sales}
      xKey="month" xScaleType="band"
      yKey="revenue" yScaleType="linear"
    >
      <Bar />                          -- inherits yKey="revenue"
      <Line y={(d) => d.runningAvg} /> -- overrides for this shape
    </ChartContainer>

The bars come from `revenue`. The line comes from `runningAvg`.
Same rows, different columns. The y-scale still spans the
revenue domain — which is fine here because runningAvg fits
inside that range. If it didn't, the line would clip; we'd
need a manual `yDomain` prop on the container. Not implemented
yet; add when a real chart needs it.

--- The shared helper: `resolveYAccessor` ---------------------

Rather than repeat the precedence logic in four files, we
factored it into `src/shapes/accessors.ts`:

    export interface YOverride<T> {
      y?: (d: T) => number;
      yKey?: KeysOfType<T, number>;
    }

    export function resolveYAccessor<T>(
      props: YOverride<T>,
      fallback: (d: T) => number | Date | string,
    ): (d: T) => number {
      if (props.y) return props.y;
      if (props.yKey !== undefined) {
        const key = props.yKey as keyof T;
        return (d: T) => d[key] as number;
      }
      return fallback as (d: T) => number;
    }

Each shape's props type EXTENDS YOverride<T> so consumers see
both new props in autocomplete. Inside each component, one
line changes:

    const yFn = resolveYAccessor<T>(props, ctx.yAccessor);
    // ...then use yFn(datum) instead of ctx.yAccessor(datum)

That's the whole edit. Line, Area, Bar, Scatter each got the
same two-line change. No context changes, no tests broken.

--- Type-safe `yKey`: KeysOfType at work ----------------------

The `yKey` prop is typed as `KeysOfType<T, number>` — a utility
we defined back in Phase 2. It's a mapped type that filters
`keyof T` to only the keys whose values extend `number`. So for
`type Sale = { month: string; revenue: number; runningAvg: number }`,
`KeysOfType<Sale, number>` resolves to `"revenue" | "runningAvg"`
— NOT `"month"`. Consumers get autocomplete showing only the
numeric fields, and passing `yKey="month"` becomes a compile
error.

This is the payoff for putting KeysOfType in place early: every
per-shape override in future phases can reuse it for typed key
props (`sizeKey` for Bubble, `groupKey` for Stacked, etc.).

4g. PHASE 5e — WHAT WE ACTUALLY DID (STACKED CHARTS)
------------------------------------------------------------

Final slice of the Cartesian shape family. Shipped two new
components — `<StackedBar>` and `<StackedArea>` — plus a small
change to `useChartScales` that unlocks them.

--- Why stacked charts need real math ------------------------

At first glance a stacked bar chart looks like a composed chart:
several series on the same x-axis. But there's a critical
difference. In a composed chart each series is drawn INDEPENDENTLY
from the baseline (y=0). In a stacked chart, each series is drawn
FROM WHERE THE SERIES BELOW IT ENDED.

Example: monthly revenue by product category
    { month: "Jan", laptops: 120, phones: 80, tablets: 45 }

Composed rendering (with three <Bar> shapes):
    laptops bar: from 0 to 120
    phones bar:  from 0 to 80  (overlaps laptops!)
    tablets bar: from 0 to 45  (overlaps both!)
Result: visual chaos.

Stacked rendering:
    laptops bar: from 0 to 120       (0-120)
    phones bar:  from 120 to 200     (120-200)
    tablets bar: from 200 to 245     (200-245)
Result: one bar per month, split into color-coded sections
showing total AND per-category contribution.

The math — "add each row's key to the running total, use the
totals to compute segment tops and bottoms" — is exactly what
`d3-shape.stack()` does for us. We wrap it.

--- The one thing that needed a REAL change: yDomain ---------

There was one architectural change this phase forced. All prior
shapes plotted against a y-scale whose domain came from
`extent(data, yAccessor)`. That works for single-series charts:
the scale spans from the min to the max of the plotted values.

Stacked charts break this. If you're stacking laptops (max 180),
phones (max 130), tablets (max 80), the auto-computed domain is
`[0, 180]` — the max of any SINGLE key. But the visible chart
needs to reach `[0, 390]` (the max SUM). Otherwise the top of the
stack renders above the chart's top edge and gets clipped.

Two ways to solve this:
  1. Consumer sets the y-domain manually.
  2. Stacked components signal a "domain hint" back to the
     container so it can auto-compute.

We picked #1 — added a `yDomain?: [number, number]` prop to
ChartContainer. If set, use it; else keep the current behavior.
Trade-off: consumers of `<StackedBar>` MUST set yDomain manually.
Not zero-config, but explicit and simple. #2 would be more
elegant but requires a two-pass architecture (children signal up,
then container re-computes). Not worth it yet.

To make the manual computation one line, we exported a
`stackSum(data, keys)` helper:

    <ChartContainer
      data={quarter}
      xKey="month" xScaleType="band"
      yKey="laptops" yScaleType="linear"
      yDomain={[0, stackSum(quarter, ["laptops", "phones", "tablets"])]}
    >
      <StackedBar keys={["laptops", "phones", "tablets"]} colors={COLORS} />
    </ChartContainer>

The `yKey="laptops"` is redundant for the stacked bar, but
required by the container's discriminated-union types — the
container doesn't know its children are stacked. Documented.

--- StackedBar & StackedArea implementation -------------------

Both components follow the same pattern:

  1. Build a stack generator: `d3stack().keys(props.keys)`
  2. Run it on data: `series = stackGen(data)` — returns an
     array-of-arrays. Each outer element is one key's series.
     Each inner element is `[y0, y1]` in data units, with the
     original data row attached as `.data`.
  3. Convert each `[y0, y1]` through `yScale` to get pixels.
  4. Render:
       StackedBar: <rect x y width height> per segment
       StackedArea: <path d> per series

One aesthetic detail on StackedBar: only the TOP segment gets
rounded corners. If ALL segments were rounded, you'd see
visible curved gaps between colors within a bar. Real-world
stacked bars round the whole bar's top (visually = top segment
only). We do that:

    rx={isTopSegment ? radius : 0}

4h. PHASE 6 — WHAT WE ACTUALLY DID (POLAR + PIE)
------------------------------------------------------------

First non-Cartesian chart family. Everything before Phase 6
assumed (x, y) → (pixel, pixel) plotting on a rectangular plot
area. Pie charts don't fit that model. They compute:
  - ANGULAR sweeps around a center point (each slice's start
    and end angle)
  - RADIAL distances from that center (inner radius for donuts,
    outer radius for the visible edge)

Data isn't x/y — it's "value per row", where the ANGULAR sweep
of each slice is proportional to that value's share of the total.

--- The architectural decision: separate containers ----------

Locked back in CLAUDE.md's initial planning: `ChartContainer`
handles Cartesian; a new `PolarChartContainer` handles polar.
NO union-typed shared context. Two reasons:

  1. The context shapes are fundamentally different. Cartesian
     needs xScale/yScale/xAccessor/yAccessor. Polar needs
     cx/cy/radius/innerRadius/valueAccessor. Trying to unify
     forces every child component to narrow a discriminated
     union just to read basic props.
  2. Cross-family shape mixing (Pie inside a Cartesian
     container?) doesn't make sense. Two separate containers
     make the "wrong shape in wrong container" case a compile
     error, not a runtime bug.

So we shipped:
  - `<PolarChartContainer>` — the new component, alongside the
    existing Cartesian `<ChartContainer>`
  - A new context (`PolarContext`) with its own hook
    (`usePolarChartContext`)
  - `<Pie>` — reads polar context
  - `<PolarTooltip>` — reads polar context

Zero code touched in the Cartesian pipeline. Fully additive.

--- What the polar container gives you -----------------------

`<PolarChartContainer>` measures its DOM size (via the same
`useResize` hook the Cartesian container uses — reused, not
duplicated), and derives three things:

    cx = width / 2
    cy = height / 2
    radius = min(width, height) / 2 - padding

That's it. No scales in the classical sense. Consumer only
supplies data + a value accessor.

Consumer surface:

    <PolarChartContainer
      data={products}
      valueKey="revenue"      // OR value={(d) => d.revenue}
      innerRadius={70}        // 0 = pie, >0 = donut
      padding={16}            // default 20; gap from card edge
    >
      <Pie colors={COLORS} label={...} />
      <PolarTooltip>{({ datum }) => <g>...</g>}</PolarTooltip>
    </PolarChartContainer>

The `valueKey` prop is typed as `KeysOfType<T, number>` — same
utility from Phase 2 that powers per-shape yKey overrides. If
you pass a string-valued key, it's a compile error.

--- How Pie draws ------------------------------------------

We use two functions from d3-shape:

  1. `pie().value(fn)` — takes an array of data, returns arc
     "data" objects with `.startAngle` and `.endAngle` fields
     already computed as radians. Handles the "share of total"
     math internally.
  2. `arc().innerRadius(r0).outerRadius(r1)` — converts an arc
     data object into an SVG path `d` string.

Composed together:

    const pieGen = d3.pie().value(v => valueAccessor(v)).sort(null);
    const arcs = pieGen(data);                      // sweep angles
    const arcGen = d3.arc()
      .innerRadius(innerRadius)
      .outerRadius(radius);
    arcs.forEach(a => arcGen(a));                    // "d" strings

`.sort(null)` is important — by default d3.pie() sorts slices by
value (largest first). We want the visual order to match the
data order the consumer provided.

The `<Pie>` component wraps each arc in a `<path>` with
`onMouseEnter` / `onMouseLeave` handlers that push the active
slice index into the polar context. That drives the tooltip.

--- The centroid label pattern -------------------------------

Every slice has a "centroid" — the pixel point at the visual
middle of the wedge. d3.arc().centroid(arc) returns it as
`[x, y]`. We pass it to consumers through the `label` render-prop:

    <Pie label={({ centroid, datum, index, cx, cy, startAngle,
                    endAngle }) => (
      <text x={centroid[0]} y={centroid[1]} textAnchor="middle" dy="0.32em">
        {formatSlice(datum)}
      </text>
    )} />

Small subtlety: the label renders INSIDE the `<g transform=
"translate(cx, cy)">` wrapper that already positions the Pie at
the container's center. So the centroid you get in the callback
is RELATIVE to the pie center, not absolute SVG coords. That's
usually what consumers want (offsets from center are more
intuitive than absolute pixel positions), but breakdown.txt is
the right place to document it.

--- Slice hover → tooltip via context ------------------------

Cartesian's `<Tooltip>` uses d3-array's bisector to find the
nearest data point to a mouse position. Polar doesn't need that —
each slice IS a hit target. So we took a simpler approach:

  1. `PolarChartContainer` keeps `activeIndex` state.
  2. `<Pie>` binds onMouseEnter/onMouseLeave to each slice
     `<path>`, calling `setActiveIndex(i)` / `setActiveIndex(null)`.
  3. `<PolarTooltip>` reads `activeIndex`; if null it renders
     nothing, otherwise it calls the children function with
     `{ datum, index, cx, cy, radius, innerRadius }`.

The consumer's render function positions the tooltip. It can
center in the pie's middle (using cx/cy), place beside the
active slice, or something else entirely. The playground uses a
centered `<foreignObject>` card because our first pie has a
donut hole that gives an obvious place for the tooltip to sit.

--- One design detail: labels use SVG, tooltip uses HTML ------

The pie's centroid label is plain SVG `<text>` inside the pie's
`<g>` wrapper. The tooltip, on the other hand, uses
`<foreignObject>` to embed an HTML card. Why the mix?

  - Labels need to sit AT slice centroids inside the pie's
    coordinate system. SVG is the natural fit.
  - Tooltips typically want richer content (multiple lines,
    colored dots, mini bar charts) that HTML+CSS handle better
    than SVG text. `<foreignObject>` is the SVG element that
    "embeds" a chunk of HTML into an SVG position.

Both are valid patterns. Consumers can use either in either
place. Playground shows both to make the choice tangible.

4i. PHASE 7 — WHAT WE ACTUALLY DID (RADAR)
------------------------------------------------------------

Final chart type in the roadmap. Radar shows a single row across
multiple numeric dimensions — think of a video-game character's
skill profile (speed / power / defense / magic / agility). Each
dimension gets its own axis radiating from a shared center. The
row's values are plotted as distances along their respective
axes and connected into a closed polygon. Two or more rows can
overlay on the same axes for comparison.

Radar reuses `<PolarChartContainer>` from Phase 6. That was the
whole reason we separated polar into its own container: the pie
and radar share a coordinate system (center + radius), the
plumbing that makes it responsive (ResizeObserver), and the
tooltip infrastructure (activeIndex hover state). All of that
came for free this phase.

--- What's genuinely new in Phase 7 ---------------------------

Radar's DATA MODEL differs from pie's. Pie takes one number per
row — the slice size. Radar takes N numbers per row — one per
axis. So Radar owns its own scale math instead of borrowing the
container's `valueAccessor`.

Two tiny math functions carry all of it:

    axisAngle(i, N) = -π/2 + (2π · i) / N
      → angle of the i-th axis. Starts at 12 o'clock (-π/2)
        and steps clockwise around the circle.

    polarPoint(angle, value, maxValue, radius)
      → (x, y) offset from center. r = (value/maxValue) * radius,
        then (x, y) = (r·cos(angle), r·sin(angle)).

Both live in `polar/radarScales.ts`, separate from the components,
so we can unit-test them without needing to render anything.

--- Small container extension ---------------------------------

Phase 6's `<PolarChartContainer>` required `valueKey` or `value`
(for pie). Radar doesn't want either — its `<Radar>` reads
multiple columns via the `axes` prop, not one column via the
container. So we relaxed the container's discriminated union to
allow all three states:

    valueKey set  → pie-flavored
    value  set    → pie-flavored (function form)
    NEITHER set   → radar-flavored (accessor is a no-op)

Existing pie users unchanged. Radar users just don't pass the
value/valueKey props. Documented but not enforced — sensible
default for a family that has two data models.

--- Three components + one helper file ------------------------

`<RadarGrid axesCount={5} rings={4} />`
    Background rings. Concentric closed polygons (default) or
    concentric circles (`shape="circle"`). No data-dependent
    logic — pure decoration derived from the axis count and
    the container's radius.

`<RadarAxes axes={["speed", "power", ...]} />`
    Spokes from center to radius along each axis, plus axis
    labels just beyond the endpoints. `textAnchor` flips
    (start/end/middle) based on the label's angular position
    so text doesn't crowd the center.

`<Radar axes={[...]} maxValue={100} />`
    The polygon(s). One `<path>` per data row, closed with Z.
    Optional `showPoints` dots at each axis intersection.
    Uses `maxValue` if set, else auto-computes the maximum from
    the data.

`polar/radarScales.ts`
    The two pure math functions plus `polygonPath(points)`,
    which formats an array of [x, y] pairs into an SVG "M x,y L
    x,y ... Z" string. Extracted so it can be tested in isolation
    (and reused by anything else needing polygon paths later).

--- Multi-series pattern (rowIndex) ---------------------------

Two radars on one chart (Alice + Bob) is the classic use case.
Rather than having `<Radar>` render ALL rows and giving the
container multiple color schemes to sort out, each `<Radar>`
takes an optional `rowIndex` prop — when set, it renders ONLY
that row. Multi-series is then just:

    <Radar rowIndex={0} fill="blue" />
    <Radar rowIndex={1} fill="emerald" />

Two radars, two colors, two polygons. Zero context extension
needed. Same pattern as putting two `<Bar>` in a Cartesian
container each with a different `y` accessor — different
component, same "each shape picks its slice" mental model.

--- Tooltip integration: zero code changes --------------------

Phase 6's `<PolarTooltip>` reads `activeIndex` from context.
When a `<Radar>` polygon fires onMouseEnter, it sets the
active index. `<PolarTooltip>` sees the change and renders the
active row's data. Hovering Alice's polygon lights up the
tooltip showing Alice's stats. Hovering Bob's lights up Bob's.

Works because we made hover state PART OF THE CONTAINER back in
Phase 6, not part of Pie. Anticipated Radar would need the same
mechanism. Paid off.

7. VOCAB CHEAT SHEET
------------------------------------------------------------

MONOREPO       — one git repo holding multiple related packages
WORKSPACES     — npm's feature for managing a monorepo
BUNDLER        — tool that packages source code for shipping (tsup)
ESM / CJS      — two module formats; we ship both
PEER DEPENDENCY — a package that MUST be provided by the consumer
                  (React, in our case). Not installed by us.
D.TS FILE      — TypeScript type declaration file. Ships alongside
                  .js so consumers get autocomplete.
TREE-SHAKING   — bundlers dropping unused exports from your final app
CONTEXT        — React's way to share values with descendant components
                  without prop-drilling
HOOK           — a `use*` function that lets components use React
                  features (state, effects, context, custom logic)
SCALE          — D3's word for a function that maps data to pixels
RESIZEOBSERVER — browser API that fires when a DOM element resizes
                  (how our charts stay responsive)
RENDER PROP    — a function passed as a component's prop; the
                  component calls it and renders whatever JSX it
                  returns. Our <Line label={fn} /> is a render prop.
                  Enables per-point customization by consumers with
                  full type safety.
TICK           — one of the labeled positions on an axis (0, 20, 40...
                  for a linear scale; "Jan", "Feb", "Mar" for a band
                  scale). d3 scales know how to generate sensible
                  tick values via `.ticks(n)`.
TICKFORMATTER  — a function you pass to XAxis/YAxis that formats each
                  tick's TEXT display, e.g. (v) => `$${v}` prints
                  "$100" instead of "100".
PLAYGROUND     — a dev-only mini-app inside our repo (packages/
                  chartly/playground/) that boots via Vite and lets
                  us see the library rendering in a real browser.
                  Not published to npm.
BISECTOR       — d3-array's binary search helper. Given a sorted
                  array and a query value, returns the index of the
                  nearest element in O(log n) instead of O(n).
                  Powers our tooltip's nearest-point lookup so it
                  stays fast on large datasets.
INVERT         — a scale's reverse operation: given a PIXEL, return
                  the DATA value it maps to. Linear/time scales have
                  it; band scales don't (there's no meaningful
                  inverse for a category). We use it to translate
                  mouse position back into "what data value is the
                  mouse hovering?"
POINTER EVENTS — the CSS/SVG property that controls whether an
                  element receives mouse/touch events. Setting
                  `pointerEvents="none"` on our tooltip content
                  ensures the invisible overlay rect underneath
                  still catches events, keeping onMouseLeave
                  reliable.
CURVE FACTORY  — d3-shape's building block for interpolating a
                  line between points. Different factories produce
                  different visual results (straight vs monotone
                  smooth vs staircase). Our CurveType prop maps
                  a string like "monotone" to `curveMonotoneX`.
BASELINE       — for an area chart, the y coordinate the fill
                  extends DOWN to. Our Area uses `height -
                  margin.bottom` (the bottom of the plot area).
                  Not the data's y=0, which would put the fill
                  off-screen for datasets whose values don't
                  include zero.
SVG DEFS       — the `<defs>` block inside an SVG holds reusable
                  definitions (gradients, patterns, filters) that
                  the rest of the SVG references by URL. Our
                  LinearGradient renders one of these; consumers
                  reference it via `fill="url(#id)"`.
BAND SCALE     — d3's scale for categorical x-axes. Domain is a
                  list of category strings ("Widgets", "Gadgets",
                  ...). Range is divided into equal-width "bands"
                  with optional padding between them. Bar uses
                  `bandwidth()` to size each rectangle.
BASELINE (BAR) — the y data value bars start FROM (default 0).
                  We CLAMP this to the plot area so revenue-style
                  data (all positive, y-domain [120, 270]) still
                  looks right — bars visually stand on the plot
                  floor without the consumer having to override
                  their y-domain.
BANDWIDTH      — the pixel width of one band in a band scale.
                  `xScale.bandwidth()` returns the same number for
                  every band, which is why all bars in a bar
                  chart naturally share a width.
SCATTER PLOT   — a chart of discrete (x, y) points as unconnected
                  dots. Good for showing correlation, spread, or
                  overlaying markers on top of a line chart.
                  Chartly's <Scatter> renders one <circle> per
                  data row.
COMPOSED CHART — a chart that shows more than one series at once,
                  usually of different shape types (bars +
                  overlay line, area + scatter markers, etc.).
                  Chartly does the SAME-DATA flavor: every shape
                  reads from the container's data array; each
                  shape picks which COLUMN to plot via its own
                  `y` or `yKey` prop.
Y ACCESSOR     — a function or key that tells a shape which value
  OVERRIDE       to pull from each data row as the "y" value. Set
                  via `y={fn}` or `yKey="fieldName"` per shape.
                  Falls back to the container's yKey if omitted.
KEYSOFTYPE     — a TypeScript utility (in src/types.ts) that,
                  given an object type T and a value type V,
                  produces the union of keys of T whose values
                  match V. We use `KeysOfType<T, number>` to
                  constrain `yKey` to numeric fields only, so
                  autocomplete never shows string-valued keys.
STACKED CHART  — a chart where multiple series are drawn on top of
                  each other (accumulated), so each stack shows both
                  the total AND per-category contribution. Different
                  from a COMPOSED chart, where series are independent
                  and can overlap.
STACK          — d3-shape's transform that takes a flat table
  TRANSFORM      (rows × columns) and produces per-row [y0, y1]
                  tuples per key, computing the running total for
                  us. Chartly wraps it in `<StackedBar>` and
                  `<StackedArea>`.
Y-DOMAIN       — the range of data values a y-scale spans. Normally
                  auto-computed from `extent(data, yAccessor)`. For
                  stacked charts, this default underreports the max
                  (any single series max, not the sum), so
                  ChartContainer accepts a `yDomain` override.
                  Use `stackSum(data, keys)` to compute it in one
                  line.
POLAR CHART    — any chart drawn using angles and radii instead of
                  Cartesian x/y coordinates. Pie, donut, and radar
                  are all polar. Chartly uses a separate
                  `<PolarChartContainer>` for these; the Cartesian
                  container's scale system doesn't apply.
CENTROID       — the visual "middle" of a shape. For a pie slice
                  (an arc-shaped wedge), the centroid is the point
                  that visually centers a label inside the wedge.
                  d3's arc().centroid(arc) computes it as [x, y].
                  Chartly passes this to the Pie `label` render-prop.
DONUT          — a pie chart with a hole in the middle. Achieved by
                  setting an inner radius greater than 0. Same
                  component as pie (`<Pie>`) with the `innerRadius`
                  prop on the container.
ARC / SWEEP    — an arc-shaped wedge of a circle. In SVG paths,
                  drawn with the "A" command. d3-shape's arc()
                  function turns a start/end angle + inner/outer
                  radius pair into that "A" command string.
FOREIGNOBJECT  — an SVG element that lets you embed a chunk of HTML
                  inside an SVG at a specific position. We use it
                  in the pie tooltip because HTML+CSS handles multi-
                  line styled content (borders, padding, rounded
                  corners) more cleanly than SVG `<text>`.
